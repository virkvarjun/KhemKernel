Give KhemKernel a molecule as a SMILES string and it returns the molecule's systematic IUPAC name. `CC(=O)Oc1ccccc1C(=O)O` comes back as `2-acetyloxybenzoic acid`, which is aspirin. On held out molecules it gets the exact structure right 95.8% of the time when it is allowed to propose several names and check them. The model is a transformer written by hand, the GPU kernels under it are written by hand, and the last push in accuracy comes from the fact that the answers are checkable.

## The pipeline, end to end

A molecule goes in as text, gets tokenized and embedded, runs through an encoder, the decoder writes a reasoning trace token by token while attending back to the encoder, and the name inside that trace gets parsed back into a molecule and compared to the input. That last step is both the eval metric and, at inference, the thing that picks the right candidate.

## Architecture at a glance

Standard encoder decoder transformer, pre-norm, with a few specifics that matter.

| Field | Value |
|---|---|
| d_model | 512 |
| heads | 8 (head dim 64) |
| d_ff | 2048, GeLU (tanh approximation) |
| layers | 3 encoder, 3 decoder |
| norm | pre-LN (LayerNorm before each sublayer, residual around it) |
| positions | learned embeddings, added to token embeddings |
| output | tied with the decoder token embedding |
| context | src up to 100 tokens, tgt up to 64 |
| vocab | 341 SMILES, 4000 IUPAC (byte pair) |
| params | about 24M |
| optimizer | Adam (0.9, 0.999, 1e-8), warmup 2000 then cosine to 1e-5, peak 2e-4 |
| training | 120k steps, batch 32, about 12.6h on one GPU |

### The blocks

Both block types are pre-norm: normalize, run the sublayer, add the residual. The encoder block has two sublayers, self-attention then a feed forward network. The decoder block has three: masked self-attention, cross-attention into the encoder, then the feed forward network. A final LayerNorm sits before the output projection.

<Figure src="/images/diagram-blocks.svg" wide alt="Pre-norm encoder and decoder block structure with residual streams" caption="Pre-norm residual blocks. The vertical line is the residual stream; each sublayer normalizes a copy, transforms it, and adds back. The decoder's middle sublayer pulls keys and values from the encoder." />

### Attention, concretely

Each attention sublayer projects its input to queries, keys, and values with separate weight matrices, splits them into 8 heads of dimension 64, and computes, per head,

```
Attention(Q, K, V) = softmax( Q K^T / sqrt(64)  +  mask ) V
```

The heads concatenate back to 512 and pass through an output projection. The mask is what distinguishes the three uses: encoder self-attention masks only padding, decoder self-attention adds a causal mask so position t cannot see the future, and cross-attention masks the encoder's padding. Every one of these, forward and backward, is a hand-written kernel checked against finite differences.

<Figure src="/images/diagram-attention.svg" wide alt="Multi-head attention internals" caption="One multi-head attention sublayer. Self-attention draws Q, K, V from the same input; cross-attention takes Q from the decoder and K, V from the encoder." />

## The trace target

The decoder does not write the name directly. It writes a structured trace, and the name is the last field:

```
<parent>benzene</parent><groups>phenol</groups><atoms>7</atoms><rings>1</rings><name>phenol</name>
```

These traces are built for every training molecule by RDKit: parent ring system or longest chain, functional groups, heavy atom count, ring count, then the name. Naming the scaffold before committing to a name gives the output a fixed structure to parse, and a readable window into what the model believed about the molecule. The tags and the group separator are reserved tokens the tokenizer never splits.

## Tokenizing names: the byte pair fix

SMILES tokenizes with the Schwaller regex, which keeps multi character atoms like `[C@@H]`, `Cl`, and `Br` whole. The whole alphabet is 341 tokens.

Names were the problem. The first model split names on word boundaries, so `acetyloxybenzoic` was one token, and any fragment seen fewer than five times in the corpus became a single unknown token. The model could not spell rare names. The fix was a byte pair tokenizer written for this, which starts from characters and repeatedly merges the most frequent adjacent pair. It reserves the special tokens and trace tags, learns 4000 merges, can always fall back to characters so it never emits an unknown on a real name, reconstructs the original string exactly, and produces shorter sequences than the word tokenizer did. Verified across the corpus: zero unknowns, zero round trip mismatches. Valid name rate went from 86% to 98%, and this single change drove most of the accuracy jump.

## The kernels, and a deliberately lopsided training loop

There are two copies of the model that share weights: a pure NumPy reference with hand-derived, finite-difference-checked backward passes, and a CUDA version. The CUDA build is a set of hand-written kernels bound to Python with pybind11: matmul (naive and tiled) and its two backward passes, batched matmul, softmax, layer norm, GeLU, cross entropy, embedding backward, Adam, bias add, transpose, and head split and merge.

The training loop is split across host and device on purpose. The transformer stack lives resident on the GPU and is updated in place by the Adam kernel, so weights do not cross the bus every step. The embedding tables stay on the host, where the gather and scatter happen in NumPy.

<Figure src="/images/diagram-train-split.svg" wide alt="Host device training split" caption="The stack never leaves the GPU; only the embedded batch goes up and the embedding gradients come back. A consequence: the GPU runs at 35 to 40% utilization, because the host side embedding work and the transfers, not the matmuls, are the bottleneck at this model size." />

One portability note from the final run. The kernels build against the GPU's compute capability, and when the GPU is newer than the toolkit they fall back to the newest virtual architecture the toolkit knows and emit PTX, which the driver compiles at load time. That is how the final model trained on a Blackwell card (compute capability 12.0) using a CUDA 12.4 toolkit that has never heard of it.

### The run that blew up

The first scaled run trained well to a loss of 0.37, then a gradient spike near peak learning rate drove the loss to NaN at step 19,000, and since the trainer overwrote a single checkpoint file the NaN weights erased the good ones. There is no gradient clipping in the device loop, and a real clip needs a reduction kernel that does not exist yet. The shipped fix is cheaper: skip the optimizer step whenever the loss is not finite, only checkpoint finite states, keep numbered snapshots, and lower the peak learning rate. The next run went the full 120k steps with zero NaN events and zero skipped steps.

## Decoding, and a verifier you get for free

OPSIN turns an IUPAC name into a molecule. The input to the model is a molecule. So the model can check its own output with no labels: generate a name, parse it back, canonicalize both sides with RDKit, ask if they are the same molecule. Instead of trusting the top guess, the decoder runs a beam, and the rerank keeps the first candidate that round trips to the input.

<Figure src="/images/diagram-rerank.svg" wide alt="Beam search plus verifier rerank" caption="Beam search proposes candidates; each is parsed back through OPSIN and compared to the input molecule; the first that matches is returned, falling back to the best parseable name, then the top beam." />

Beam scores are length normalized as `sum(log p) / len^0.6` so the search does not favor short names. The accuracy this buys, on 2,000 held out molecules:

| Decoding | First model | Final model |
|---|---|---|
| Greedy, one pass | 67.1% | 79.5% |
| Beam 5 plus verifier | 81.0% | 89.6% |
| Beam 20 plus verifier | | 95.8% |
| Valid name rate | 85.6% | 97.9% |

## Where the remaining errors are

Three diagnostics, not guesses. The first measured stereochemistry by stripping it from both prediction and target and rechecking. It is a non-issue: when the model gets a molecule's skeleton right it gets the stereochemistry right too, 887 times out of 896, so pure stereo mistakes are under one point of the gap. The real errors are constitutional, right atoms in the wrong arrangement, and about half of them share the exact molecular formula of the target, which is the signature of a positional isomer.

The second measured headroom: a correct answer is in the top 20 candidates for 95.8% of molecules, and the curve is still climbing at 20. The model usually knows the answer and ranks it just below first, which is exactly why the verifier rerank works, and why the deployed accuracy already sits at that ceiling. Pushing past it means improving the model's view of the molecular graph rather than its decoding, which is a research build, so this is where the project stops.
