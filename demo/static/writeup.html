How ChemKernel names molecules

Building a SMILES to IUPAC translator from hand-written NumPy gradients up through CUDA kernels, a byte pair tokenizer, and a verifier that lets the model check its own answers. The story of getting from 67% to 95.8% exact match.

You can hand ChemKernel a molecule written as a SMILES string and it will write back the molecule's systematic IUPAC name. Type in CC(=O)Oc1ccccc1C(=O)O and it answers 2-acetyloxybenzoic acid, which is aspirin. Type in ibuprofen and it gives you 2-[4-(2-methylpropyl)phenyl]propanoic acid. On molecules it has never seen it gets the exact structure right 95.8% of the time.

That last number is the one I care about, and most of this post is the story of how it got there, from the gradients up. The short version: the model is a transformer written by hand, the GPU kernels under it are also written by hand, and the final push in accuracy came from noticing that the model can check its own answers for free.

Why naming molecules is a good problem

I picked SMILES to IUPAC translation because it sits in a useful spot. There is a clean ground truth, so you are never arguing about whether an answer is right. The vocabulary is small enough that full attention is cheap at modest model sizes. And there is a long tail of genuinely hard cases, like stereochemistry, fused ring systems, and where exactly a substituent attaches, so the problem does not collapse the moment you start.

The part I did not appreciate at the start, and which ended up mattering most, is that the answer is checkable. There is an open source parser called OPSIN that reads an IUPAC name and gives you back the molecule. So if the model proposes a name, I can parse it back into a structure and compare it to the molecule I started with. No labels needed at inference, just a yes or no on every guess. Hold that thought.

The shape of the thing

The model is an ordinary encoder decoder transformer. The encoder reads the SMILES tokens. The decoder writes the answer one token at a time and looks back at the encoder through cross attention. Nothing exotic.

The one design choice worth calling out is that the model does not write the name directly. It writes a short trace first:

<parent>benzene</parent><groups>phenol</groups><atoms>7</atoms><rings>1</rings><name>phenol</name>

It names the parent scaffold, lists the functional groups, counts the heavy atoms and rings, and only then commits to a name. I build these traces for every training molecule with RDKit. Two reasons. It makes the output trivial to parse and check, since the answer is always inside the <name> tag. And it gives somewhere to look when you want to know what the model believed about a molecule before it named it.

Built twice, on purpose

There are two complete copies of this model. One is pure NumPy, where every forward pass has a backward pass I derived by hand and checked against finite differences in the test suite. The other is CUDA, with kernels I wrote for matmul and both of its backward passes, softmax, layer norm, GeLU, cross entropy, the embedding lookup, Adam, and so on, wired into Python through pybind11.

The NumPy version is the reference. It is slow and it is correct, and the CUDA version is validated against it. This sounds like extra work and it is, but it saved me more than once. When a kernel was wrong, the symptom was a training loss that looked plausible but learned nothing, and the only way I caught it quickly was running the same batch through both and diffing the gradients.

The CUDA trainer keeps the transformer stack resident on the device and updates it in place, so a training step is not copying weights across the bus every iteration. The embedding tables stay on the host and do their gather and scatter on the CPU. That split has a funny consequence: during training the GPU runs at about 35 to 40% utilization, because the host side embedding work and the data movement are the real bottleneck, not the matmuls. For a model this size that is fine, and it means a fancier GPU would not have helped.

Spelling is the hard part

SMILES tokenizes cleanly. There is a well known regex that keeps multi character atoms like [C@@H] and Cl intact, and the whole SMILES alphabet comes out to 341 tokens.

Names are where the first version fell down. I started by splitting IUPAC names on word boundaries, so a chunk like acetyloxybenzoic became one token. Any chunk that showed up fewer than five times across the corpus got mapped to a single unknown token. The effect was that the model could not spell anything rare. Ask the first model for alanine and it produced 2- then an unknown token then acid. It knew the shape of the answer and could not write it.

The fix was a byte pair tokenizer I wrote for this. It treats the trace tags as atomic and learns merges over everything else, starting from individual characters and combining the pairs that show up most often. The result has 4,000 tokens, can spell any name because it can always fall back to characters, reconstructs the exact original string, and produces shorter sequences than the word tokenizer did. After the swap the rate of valid, parseable names went from 86% to 98%, and alanine came back as 2-aminopropanoic acid, spelled correctly, with the space and everything.

This was the single biggest change. Greedy accuracy went from 67% to 79.5% on the strength of the tokenizer plus a wider model and a proper learning rate schedule.

The run that blew up

The scaled retrain did not go smoothly the first time. It trained well, loss dropping past 0.37, and then at step 19,000 the loss became NaN and stayed there for the remaining hundred thousand steps. A gradient spike near peak learning rate had blown the model up, and because the trainer was overwriting a single checkpoint file, the NaN weights had already erased the last good ones. The whole overnight run was scrap.

The device trainer has no gradient clipping, and clipping properly would need a reduction kernel I had not written. So I took the cheaper route. Skip the optimizer step entirely whenever the loss is not finite, so one bad batch cannot poison the weights. Only ever write a checkpoint when the loss is finite. Keep numbered snapshots instead of one overwriting file. And drop the peak learning rate a notch for margin. The next run went the full 120,000 steps with zero NaN events and, as it turned out, zero skipped steps, because the lower learning rate avoided the spike on its own. The guards never had to fire, but I am keeping them.

The model can check its own answers

Here is where the held thought pays off. Because OPSIN turns a name back into a molecule, and the input to the model is a molecule, the model can grade its own output without any label. Generate a name, parse it back, canonicalize both molecules with RDKit, and ask if they are the same thing.

So instead of trusting the model's single best guess, I have the decoder produce a beam of candidates and keep the first one that round trips back to the input molecule. A beam of five takes structure match from 79.5% to 89.6%. A beam of twenty takes it to 95.8%. The model usually has the right answer somewhere in its top guesses. It just does not always put it first, and the verifier picks it out at no training cost.

This is the kind of trick that only works when you have a cheap, exact checker. Most generation problems do not. Molecule naming does, and it would be a waste not to use it. It is also what the SMILES box on this demo is doing right now.

What it actually gets wrong

Before deciding what to do next I wanted to know precisely what the remaining errors were, so I wrote three small diagnostics rather than guess.

The first result overturned my assumption. I was sure stereochemistry was the problem, since it is fiddly and 18% of the test molecules have stereocenters. It is not. When the model gets a molecule's skeleton right, it gets the stereochemistry right too, 887 times out of 896. If you ignored stereochemistry entirely, accuracy would move by less than a point. I had been ready to spend real effort on the wrong thing, and the only reason I did not is that I measured it.

The actual errors are constitutional. Right atoms, right groups, wrong arrangement. A methyl on the wrong ring carbon, a wrong locant, a wrong ring fusion. Roughly half of the remaining mistakes have the exact same molecular formula as the target, which is the fingerprint of a positional isomer: all the right pieces, assembled wrong.

The third diagnostic measured how much the model already knows but ranks low. For 95.8% of molecules a correct answer is somewhere in the top twenty candidates, and the curve was still rising at twenty. So the knowledge is mostly there. The bottleneck is ranking, not capability.

Would reinforcement learning help, and why I stopped

That last finding is exactly the setup where reinforcement learning, or its simpler cousin rejection sampling, tends to help: the model can produce the right answer, it just does not rank it first, and there is a perfect verifier to score attempts. You sample, keep the verified answers, and fine tune on them so the model learns to put the good answer first.

I thought hard about doing it and decided against it for now, for an honest reason. The verifier rerank already gets the deployed model to 95.8%, which is essentially the same ceiling the model's own candidates set. Reinforcement learning would mostly convert that into a faster single pass model, not a more accurate one, and the project already has ground truth labels, which makes the technique less of a slam dunk than the raw headroom number suggests. The genuine next gain in accuracy lives in the constitutional errors, and fixing those means giving the model a better view of the molecular graph, which is a real research build rather than a quick win.

So this is a good place to stop. The model names molecules correctly 95.8% of the time, it runs on a laptop, the demo lets you try either direction, and I know exactly where the remaining five percent goes.
